Mockup notes (not rendered in final plugin):
· Scale model (same as Remove tool): plugin reads a CSV of devices, silently reads each device's port scope, groups devices by fingerprint, presents ONE prompt per unique group.
· Inverted focus: where Remove surfaces in-scope ports so operator can trim them, Add surfaces out-of-scope ports so operator can provision them. The "In Scope" column is the main new element; isActive=false ports are the interesting rows.
· Default filter hides in-scope ports (typically the majority). "Show in-scope" toggle reveals them with a muted, read-only visual treatment (checkbox disabled).
· Non-destructive. Adding a port provisions fresh agent resources and begins metric collection. No data loss risk; no typed-confirmation gate needed downstream. The destructive-banner from the Remove tool is replaced with a green informational banner.
· Accent color for the ADD row-highlight is green (--ok) to visually distinguish the additive action from the destructive orange used by Remove.
· fortilink still carries a fabric-link tag if operator somehow gets it out-of-scope — same "verify before acting" pattern as Remove.
· "Queue for N devices" stages the decision in chrome.storage.local. Nothing writes to FortiMonitor until queue execution (step 4).
· Same underlying API as Remove (save_port_selection) — the mechanism only differs in which port indices appear in selectedPorts[].
