Mockup notes (not rendered in the plugin):
· Bulk by design. Mirrors FMN-45: paste-a-list input, one shared operation, per-row preview, explicit Execute. Single-server is just N=1.
· Two operations in one tool: Add/Set (with a value) or Remove (type only, value field hidden). Segmented control at the top.
· REPLACE semantic: when a server already has a different value for the chosen type, the tool deletes the old value then POSTs the new one. This depends on the live-probe open question in docs/api-discovery/attributes.md — if POST-with-existing-type actually replaces on the server, the DELETE becomes unnecessary; if it stacks, REPLACE is correct; if it errors, REPLACE is required. Pick the safe path and verify on the first real run.
· ALREADY SET skip: rows where the current value equals the new value are shown but skipped, so the operator can see the full intended state without churn.
· Unresolved names surface in the preview, not silently. Same name→id resolver as FMN-50 will handle this — reuse, don't reinvent.
· System attributes (server.origin, server.os): not pickable in the type dropdown for Remove — FortiMonitor repopulates them from the agent, so operating on them is pointless.
· No queue, no typed confirmation: low blast radius (attribute metadata, not port scope or monitoring config). Preview + Execute is sufficient.
· Error handling post-Execute: step 4 (Results) shows per-server success/fail with status code and body. Partial failures don't abort the batch.
