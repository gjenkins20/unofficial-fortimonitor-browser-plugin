Mockup notes (not rendered in final plugin):
· Step 3 of 4. Previous step is interface-report.html; next is the execution progress screen.
· Five mock groups shown: 4 have pending removals (79 devices), 1 was skipped by the operator (8 devices). Total batch: 87 devices.
· Group cards are collapsible. Group 1 is shown expanded as the reference pattern. Expanded view shows ports marked for removal, ports kept, and a device sample with "+N more" overflow.
· "Edit this group" returns the operator to the interface-report for that specific group; their prior selection is preserved so they can modify rather than start over.
· Destructive gate: dry-run is ON by default. Primary button reads "Run dry run →" until dry-run is off. When dry-run is off, operator must type the exact phrase EXECUTE N PORTS (N matches the live count) into the confirmation input before the primary button (then "Execute and remove N ports") enables. This is a soft guard against bulk accidents.
· Plan export: JSON and CSV downloads are offered before execution since the action is irreversible. The export is the durable record of what was done.
· Queue state is kept in chrome.storage.local so it survives tab navigation. Executing the queue replays the same Port Selection save that the FortiMonitor WebGUI dialog performs when the operator clicks Save — once per device, using the existing logged-in session. No separate credentials, no external system.
· Execution modes: default is silent background execution (no visible tab navigation, concurrency up to 3). Verbose mode opts into visibly navigating a dedicated tab through each device serially so the operator can watch each change — useful for spot-checking early in a batch, too slow for routine runs at fleet scale.
