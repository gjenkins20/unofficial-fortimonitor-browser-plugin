Mockup notes (not rendered in final plugin):
· Step 3 of 4. Previous step is tool-add-port-scope_review.html; next is the execution progress screen.
· Five mock groups shown: 4 have pending additions (79 devices), 1 was skipped by the operator (8 devices). Total batch: 87 devices.
· Group cards are collapsible. Group 1 is shown expanded as the reference pattern. Expanded view shows ports marked to add, ports already in scope, and a device sample with "+N more" overflow.
· "Edit this group" returns the operator to the review screen for that specific group; their prior selection is preserved so they can modify rather than start over.
· No typed-confirmation gate. The Remove tool requires operator to type EXECUTE N PORTS because deletion is irreversible. Adding ports is non-destructive — reversible via the Remove tool — so dry-run (on by default) is the only guard. Primary button reads "Run dry run →" until dry-run is off; then it becomes "Add N ports" (green, no typed gate).
· Plan export: JSON and CSV downloads offered so the operator still has an audit trail even though the action is reversible.
· Queue state is kept in chrome.storage.local so it survives tab navigation. Executing the queue replays the same save_port_selection that the FortiMonitor Port Selection dialog performs when the operator clicks Save manually — once per device, using the existing logged-in session. No separate credentials, no external system.
· Execution modes: default is silent background execution (no visible tab navigation, concurrency up to 3). Verbose mode opts into visibly navigating a dedicated tab through each device serially so the operator can watch each change — useful for spot-checking, too slow for routine runs at fleet scale.
