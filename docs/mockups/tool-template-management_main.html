Mockup notes (not rendered in the plugin):
· Two modes, one tool. Attach and Detach share the start → preview → execute → results flow. The segmented control at the top is the mode switch; the strategy radio only appears in Detach mode.
· Strategy radio is the critical UX of this tool. v2 DELETE accepts {strategy: "dissociate"|"delete"}. Default to dissociate — the safe path. "Delete" is visually marked danger and requires typed confirmation.
· Typed confirmation gate. Shown when: (a) Detach with strategy=delete, OR (b) Attach to >10 servers. Confirmation phrase differs by operation so muscle memory doesn't cross-apply: ATTACH N SERVERS / DETACH N SERVERS / DELETE METRICS.
· Pre-flight is mandatory. Before the preview table renders, the background issues one GET /v2/server/{id}/template per resolved target to decide attach-vs-skip and detach-vs-skip. This is concurrency-budgeted the same way execute is (default 4).
· NOT ATTACHED skip (Detach mode). Servers without the template attached are shown but skipped so the operator sees the full intended state. Same shape as FMN-48's ALREADY SET skip.
· ALREADY ATTACHED skip (Attach mode, not shown). Symmetric case: if the template is already attached to a server, we show it with a skip pill rather than POST a duplicate mapping. Prevents the duplicate-mapping footgun the schema allows.
· Name resolution is FMN-50. Reuse lookupServersByName, case-sensitive exact match, unresolved names surface as preview errors (not silent). Same as FMN-48.
· Beta. Stays Beta until the FMN-46-style live E2E ticket passes (delete-strategy destructiveness needs to be verified against a real lab server — the schema claims it but we've never run it).
· Alert storm risk: attaching a template with service checks at scale creates real monitoring checks — misconfigured threshold = alert storm. That's why Attach at >10 servers also requires typed confirmation, even though it isn't "destructive" in the API sense.
