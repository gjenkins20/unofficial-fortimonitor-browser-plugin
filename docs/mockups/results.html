Mockup notes (not rendered in final plugin):
· Final step in the flow. Reached automatically when every device in the queue has settled into a terminal state (succeeded or failed-after-retries-exhausted).
· Verdict banner has three variants: success (all devices succeeded, green), partial (some failed, amber — shown here), fail (all failed, red). Copy and icon adjust accordingly.
· Download report is prominent because the executed changes are irreversible — the CSV/JSON is the durable record of what was done. Report includes batch ID, timing, per-device outcomes, ports removed, and error details.
· Failures section shows every device that did not succeed, with concrete error, attempt history, and an inline Retry button. Retries re-submit just that device using the current session. If the session is expired the operator must re-authenticate in FortiCloud first — the plugin does not handle SSO.
· Retry all failures runs the failed devices through the same execution pipeline as a mini-batch; results are merged into this view when done.
· Start new batch clears the current queue and returns to step 1 (batch-start.html). Close dismisses the plugin UI but preserves the batch record in chrome.storage.local so the operator can reopen it later.
· Skipped groups are reflected explicitly so the operator has a complete picture of what the batch touched vs. didn't.
· All reads and writes use the existing FortiCloud session — the same one the WebGUI uses. Nothing runs outside the browser.
