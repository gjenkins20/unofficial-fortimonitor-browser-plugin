What's happening: content script intercepts the navigation to /report/ListServers and calls location.replace() to a plugin-served HTML page. The page fetches the same data FortiMonitor's DataTables fetches (/report/server_group_inventory_data) plus per-row /report/get_idp_data using credentials:'include' - the FortiMonitor session cookie travels because host_permissions covers the origin. Renders a native HTML <table> with no DataTables, no Vue, no jQuery. Banner stays pinned at the top.

3Operator visible signals

things that should be different
  • URL bar changes from https://fortimonitor.forticloud.com/... to chrome-extension://<id>/...
  • Blue banner strip at the top with "Toolkit-served view (FMN-70)." and a Restore native button
  • FortiMonitor page chrome (left sidebar with DASH/INST/RPT, top nav bar) is GONE - the takeover owns the entire viewport
  • Table layout: 7 separate columns instead of native's "Instance / Group / Heartbeat" with sub-cells
  • Click "Restore native" → goes back to native /report/ListServers?fmn-skip-takeover=1 (toggle stays on for the next visit; only this navigation skips)

If you saw none of those after toggling on + closing/reopening the FortiMonitor tab, the most likely culprits are:

  1. The content script didn't pick up the new flag because the extension wasn't reloaded - chrome://extensions/ → reload Unofficial FortiMonitor Toolkit, then close and reopen the FortiMonitor tab.
  2. The web_accessible_resources entry didn't apply - again, an extension reload picks up manifest.json changes.
  3. You're on a /report/ListServers URL with a stale query param (?fmn-skip-takeover=1) from a prior "Restore native" click; navigate to https://fortimonitor.forticloud.com/report/ListServers clean.
  4. The redirect fired but the takeover page failed to load (e.g., cross-origin cookie not attaching). Open DevTools on the chrome-extension://<id>/... page and check the Network tab for the inventory fetch.
